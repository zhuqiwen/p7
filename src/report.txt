1. Give an English description of your solution strategy. Explain exactly how wires are connected and what you do when you are unable to make a connection.

 algorithm:
	 *
	 * Each cell is assigned a path estimate value, the sum of real cost from start to the cell and estimated cost from the cell to the end.
	 *  1. the smaller the path estimate is, the closer it is to the end. This is because the estimated cost from the cell to the end helps to guide the direction.
	 *  2. Manhattan distance is used for the estimated cost from the cell to the end.
	 *  3. In Cell class, startToHereCost is the real cost from start point to current cell; hereToEndEstimate is the distance estimate from the current cell to the end;
	 *
	 * For each current cell, we set up these attributes of its legal neighbors*, and add these neighbors into a priority queue, called potentialNexts.
	 * as long as the potentialNexts is not empty, we set current cell to be the cell with minimum path estimate value in the potentialNexts, in other words, we jump,
	 *  then we continue to set up the neighbors of the new current cell, and put the old current cell into a temporary visited list.
	 *  *1. a legal neighbor needs to be:
	 *      1. on the board
	 *      2. NOT an obstacle
	 *      3. NOT used by any path
	 *
	 * If the end cell is found in the temporary visited list, meaning a path from start to end is found, then we trace back the path, using the parent attribute.
	 * Or if the potentialNexts is empty, meaning we have tried every possibility and no path is found, we return null.
	 *
	 * NOTE: if multiple cells in the potentialNexts share the same minimum path estimate value, the last interted one should be pop out.
	 *       this ensures we don't go back to the cells with lower total path estimate but indeed much farther from the end point.
	 *       to make sure this, a custom comparator that takes the priority queue insert order of each cell is used for the potentialNexts.
	 * If no path can be found, we don't trace back, just set the path variable to be null.

2. Write a high-level pseudo-code description of your algorithm to route one wire.

 pseudo-code:
	 *
	 * currentCell <- start point
	 * while(potentialNexts is not empty){
	 *     current cell <- element with min total cost estimate in potentiaNexts
	 *     for each neighbor of current cell{
	 *         if neighbor is legal to be added to potentialNexts{
	 *             calculate its real cost from start point*
	 *             calculate its estimate cost to end point
	 *             a global insert order ++
	 *             potentialNexts.add(neighbor)
	 *         }
	 *     }
	 *     visitedCells.add(currentCell)
	 *     if(visitedCells.contains(end point)){
	 *         trace back path from end point
	 *         return path
	 *     }
	 *
	 * }
	 *
	 * return null;

3. Enumerate any data structures you use along the way and explain their purpose in your algorithm.

 class Cell
 This is a wrapper for Coord object. It comes with several attributes used to decide a coord's priority level in potentialNexts.
 It also has a parent attribute to indicate which is the previous cell, used to draw path from start to end.

 total cost estimate
 This is the sum of real cost from start point to current point and the estimate cost from current point to end point

 potentialNexts
 This is a PriorityQueue with custom comparator. The priorityQueue make sure that a cell with the minimum total cost estimate will be popped out.
 The custom comparator makes sure that when multiple cells / coords share the same minimum total cost estimate, the last inserted one will be popped out.

 locker
 This is a simple structure that is parallel with potentialNexts, to help decide if a neighbor is already in the potentialNexts.

 visitedCells
 This is a temporary list store every cell / coord that has been visited. by visited, I mean they are explored but are not set as on the final path.

 onPath
 This is a global cell / coord recorder for all wires on a chip. By maintaining this structure,
    the previous wires' paths can be recorded, to make sure when working on later wires, they will not step on a cell that is a part of another wire's path.



4. Tell us about any heuristics you've implemented and why you think they're useful.

 minimum total cost estimate, and last inserted one with minimum total cost estimate:
     *  1st, compare the pathCostEstimate value (startToCurrCost + currToEndEstimate) to push the cell with minimum pathCostEstimate to the top.
 	 *  2nd, if two cell has the same pathCostEstimate, then compare their insertOrder, the one insterted later comes first.

 	 The cell with minimum total cost estimate implies that the path finding is toward the correct direction.
 	 And last inserted rule makes sure that when multiple cells share the same minimum total cost estimate, the last inserted one, also the closer-to-end one, will be popped out.



 Updating the real cost from start for cell in the potentialNexts:
 	 * for a legal neighbor that can be added into the next potential cell list,
 	 * if the neighbor is already in the list,
 	 *  see if current cell's startToHereCost + 1 > neighbor's startToHereCost
 	 *      if so, update neighbor's startToHereCost to be current cell's startToHereCost + 1
 	 *
 	 * if the neighbor is NOT in the queue,
 	 *  calculate its attributes and add it into the queue.

 	 This is used to preventing path finder suddenly jump to the wrong direction cell far away.
